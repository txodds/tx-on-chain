## Overview

This guide demonstrates how to validate scores data against on-chain Merkle roots using cryptographic proofs. You'll learn how to fetch validation data and perform single-stat, two-stat, V2 multi-stat, and V3 multiproof validations.

The score validation endpoints support three validation shapes:

| Endpoint | Query shape | On-chain method | Notes |
|----------|-------------|-----------------|-------|
| `/api/scores/stat-validation` | `statKey=...` and optional `statKey2=...` | `validateStat` | Legacy single-stat and two-stat validation. |
| `/api/scores/stat-validation` | `statKeys=1,2,...` | `validateStatV2` | Multi-stat validation with indexed strategy predicates. |
| `/api/scores/stat-validation-v3` | `statKeys=1,2,...` (up to 5) | `validateStatV3` | Optimised multiproof validation with compressed proof data (v1.5.6+). |

Use the same network for the API request, IDL/types, program ID, and on-chain root PDA.

<Info>
The repository includes runnable devnet examples for both paths in [`examples/devnet/scripts`](https://github.com/txodds/tx-on-chain/tree/main/examples/devnet/scripts). See [Runnable Devnet Examples](/documentation/examples/devnet-examples) for setup and script mapping.
</Info>

## Validation Checklist

Before calling either validation method, confirm:

- The proof came from the same API host as your activated subscription.
- The program ID matches the network used for the proof.
- `daily_scores_roots` is derived from the same timestamp you pass into `validateStat`.
- The epoch day is encoded as u16 little-endian.
- Every proof hash is decoded to exactly 32 bytes.
- The fixture ID and sequence number refer to the score update you intend to validate.
- For V2 and V3, each strategy index refers to the same position in the requested `statKeys` array.
- The returned fixture ID and stat keys match the request, and V2/V3 response arrays have the requested length and order.
- For V3, confirm `multiproof.indices` and `multiproof.hashes` are present and non-empty.
- Every `.view()` result is exactly `true`; a thrown simulation or `false` result is a failure.

## Setup

```typescript
import * as anchor from "@coral-xyz/anchor";
import { PublicKey, ComputeBudgetProgram } from "@solana/web3.js";
import { BN } from "@coral-xyz/anchor";
import axios from "axios";
import { discoverScoreRecord } from "../common/flow";
import type { Txoracle } from "./types/txoracle"; // Use the matching network type.
import txoracleIdl from "./idl/txoracle.json"; // Use the matching network IDL.

const provider = anchor.AnchorProvider.env();
anchor.setProvider(provider);
const NETWORK: "mainnet" | "devnet" = "mainnet";
const expectedProgramId = new PublicKey(NETWORK === "mainnet"
  ? "9ExbZjAapQww1vfcisDmrngPinHTEfpjYRWMunJgcKaA"
  : "6pW64gN1s2uqjHkn1unFeEjAwJkPGHoppGvS715wyP2J");
const program = new anchor.Program<Txoracle>(txoracleIdl as Txoracle, provider);
if (!program.programId.equals(expectedProgramId)) {
  throw new Error(`Loaded IDL does not match the selected ${NETWORK} program`);
}

// Create HTTP client with authentication
const httpClient = axios.create({
  timeout: 30000,
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Bearer ${jwt}`,
    "X-Api-Token": apiToken
  },
  baseURL: NETWORK === "mainnet"
    ? "https://txline.txodds.com/api"
    : "https://txline-dev.txodds.com/api",
});

function toBytes32(value: unknown, label = "proof bytes"): number[] {
  let bytes: Uint8Array;

  if (Array.isArray(value)) {
    const elements = Array.from({ length: value.length }, (_, index) => value[index]);
    if (!elements.every((byte) =>
      Number.isInteger(byte) && byte >= 0 && byte <= 255
    )) {
      throw new Error(`${label} must contain only byte integers`);
    }
    bytes = Uint8Array.from(elements);
  } else if (value instanceof Uint8Array) {
    // Node.js Buffer is also a Uint8Array.
    bytes = Uint8Array.from(value);
  } else if (typeof value === "string" && value.startsWith("0x")) {
    const hex = value.slice(2);
    if (!/^[0-9a-fA-F]{64}$/.test(hex)) {
      throw new Error(`${label} must be canonical 0x-prefixed 32-byte hex`);
    }
    bytes = Buffer.from(hex, "hex");
  } else if (typeof value === "string") {
    if (
      value.length === 0 ||
      value.length % 4 !== 0 ||
      !/^[A-Za-z0-9+/]+={0,2}$/.test(value)
    ) {
      throw new Error(`${label} must be canonical base64 or 0x hex`);
    }
    const decoded = Buffer.from(value, "base64");
    if (decoded.toString("base64") !== value) {
      throw new Error(`${label} is not canonical base64`);
    }
    bytes = decoded;
  } else {
    throw new Error(`${label} has an unsupported encoding`);
  }

  if (bytes.length !== 32) {
    throw new Error(`${label} must be 32 bytes, received ${bytes.length}`);
  }

  return Array.from(bytes);
}

function toProofNodes(nodes: unknown) {
  if (!Array.isArray(nodes)) throw new Error("Proof must be an array");
  return nodes.map((node, index) => {
    if (!node || typeof node !== "object") {
      throw new Error(`Proof node ${index} has an invalid shape`);
    }
    const proofNode = node as Record<string, unknown>;
    if (typeof proofNode.isRightSibling !== "boolean") {
      throw new Error(`Proof node ${index} has an invalid sibling direction`);
    }
    return {
      hash: toBytes32(proofNode.hash, `proof node ${index} hash`),
      isRightSibling: proofNode.isRightSibling,
    };
  });
}
```

<Warning>
The hosted OpenAPI currently declares `ProofNode.hash`, `eventStatRoot`, and `eventStatsSubTreeRoot` as `string/binary`, while existing clients also encounter byte arrays. Until a covered live response establishes one encoding, use a strict decoder that accepts only `number[]`, `Uint8Array`/`Buffer`, canonical base64, or `0x` hex and returns exactly 32 bytes. Never use `Array.from(value)` on a string.
</Warning>

## Fetching scores data

For current records, the runnable devnet examples use `discoverScoreRecord` to scan bounded UTC update windows and query snapshot or historical data only for observed fixtures. Its `finalOnly` path instead skips current updates, discovers eligible fixture IDs from the six-hour-to-14-day fixture window, and queries those historical records. Both paths normalize `FixtureId ?? fixtureId`, `Seq ?? seq`, and `Ts ?? ts` and select stat keys that exist in the same record.

```typescript
const selected = await discoverScoreRecord(httpClient, 1);
const { fixtureId } = selected;
const response = await httpClient.get(`/scores/snapshot/${fixtureId}?asOf=${Date.now()}`);
console.log(`Retrieved snapshot records for fixture ${fixtureId}`);
```

You can still inspect a current UTC five-minute bucket directly. Derive its path from the current proof window instead of using a fixed date:

```typescript
const now = new Date();
const targetTime = new Date(now.getTime() - (5 * 300000)); // 25 minutes ago
const epochDay = Math.floor(targetTime.getTime() / 86400000);
const hourOfDay = targetTime.getUTCHours();
const interval = Math.floor(targetTime.getUTCMinutes() / 5);

const updates = await httpClient.get(
  `/scores/updates/${epochDay}/${hourOfDay}/${interval}`
);
console.log(`Retrieved ${updates.data.length} update records`);
```

## Choosing the Score Sequence

The `seq` parameter is not a placeholder. Use the sequence value from a real score record observed through one of the scores data endpoints:

- `/api/scores/snapshot/{fixtureId}`
- `/api/scores/updates/{epochDay}/{hourOfDay}/{interval}`
- `/api/scores/historical/{fixtureId}`
- `/api/scores/stream`

Do not call `/api/scores/stat-validation` with `seq=0`. Score sequences start at `1` for a fixture and increment as new score records are produced. Depending on your client and response mapper, the payload field may appear as `Seq` or `seq`; use that observed value when requesting the validation proof.

```typescript
const selection = await discoverScoreRecord(httpClient, 1);
const { fixtureId, seq, ts } = selection;
const statKey = selection.statKeys[0];

const validation = await httpClient.get("/scores/stat-validation", {
  params: {
    fixtureId,
    seq,
    statKey,
  }
});

if (
  Number(validation.data.summary.fixtureId) !== fixtureId ||
  Number(validation.data.Ts ?? validation.data.ts) !== ts ||
  Number(validation.data.statToProve.key) !== statKey
) {
  throw new Error("Legacy proof response does not match the requested record");
}
```

## Phase and Status Semantics

Pick the score record whose game phase matches the condition you want to prove. For example, "final score of the first half" should use the record for the completed first-half phase or halftime/rest state, not an arbitrary in-running first-half update.

If you validate an in-running first-half record, the predicate means the condition was true at that observed moment. That is different from proving the final first-half result. Check the sport-specific phase tables, such as [Soccer Feed](/documentation/scores/soccer-feed), and use the score record's `statusId`, `StatusId`, `gameState`, or equivalent phase field when deciding whether a record is suitable for settlement.

For final match outcome settlement on the current devnet and mainnet releases, filter `action=game_finalised` separately from ordinary score updates and require `statusId=100` plus proof `ScoreStat.period=100`. These are release-specific finalisation markers, not the sport phase IDs in the feed tables; re-verify them against the API and deployed program when adopting V3 or a newer release.

## Single-Stat Validation

Validate a single statistic against on-chain Merkle roots:

```typescript
const single = await discoverScoreRecord(httpClient, 1);
const fixtureId = single.fixtureId;
const seq = single.seq;
const statKey = single.statKeys[0];

const response = await httpClient.get("/scores/stat-validation", {
  params: { fixtureId, seq, statKey }
});
const validation = response.data;

if (
  Number(validation.summary.fixtureId) !== fixtureId ||
  Number(validation.Ts ?? validation.ts) !== single.ts ||
  Number(validation.statToProve.key) !== statKey
) {
  throw new Error("Single-stat proof does not match the requested record");
}

// Prepare fixture summary
const fixtureSummary = {
  fixtureId: new BN(validation.summary.fixtureId),
  updateStats: {
    updateCount: validation.summary.updateStats.updateCount,
    minTimestamp: new BN(validation.summary.updateStats.minTimestamp),
    maxTimestamp: new BN(validation.summary.updateStats.maxTimestamp),
  },
  eventsSubTreeRoot: toBytes32(validation.summary.eventStatsSubTreeRoot),
};

// Prepare Merkle proofs
const fixtureProof = toProofNodes(validation.subTreeProof);
const mainTreeProof = toProofNodes(validation.mainTreeProof);

// Prepare stat to validate
const stat1 = {
  statToProve: validation.statToProve,
  eventStatRoot: toBytes32(validation.eventStatRoot),
  statProof: toProofNodes(validation.statProof),
};

// Exact sanity predicate: the returned value must equal itself on-chain.
const predicate = {
  threshold: validation.statToProve.value,
  comparison: { equalTo: {} },
};

// Find the daily scores PDA
const targetTs = validation.summary.updateStats.minTimestamp;
const epochDay = Math.floor(targetTs / (24 * 60 * 60 * 1000));

const [dailyScoresPda] = PublicKey.findProgramAddressSync(
  [
    Buffer.from("daily_scores_roots"),
    new BN(epochDay).toArrayLike(Buffer, "le", 2),
  ],
  program.programId
);

// Execute validation using view (read-only simulation)
const computeBudgetIx = ComputeBudgetProgram.setComputeUnitLimit({
  units: 1_400_000
});

const isValid = await program.methods
  .validateStat(
    new BN(targetTs),
    fixtureSummary,
    fixtureProof,
    mainTreeProof,
    predicate,
    stat1,
    null,  // No second stat
    null   // No operator
  )
  .accounts({
    dailyScoresMerkleRoots: dailyScoresPda
  })
  .preInstructions([computeBudgetIx])
  .view();

if (isValid !== true) {
  throw new Error("On-chain single-stat validation returned false");
}
console.log("On-chain stat validation passed");
```

## Two-Stat Validation

Validate a comparison between two stats (e.g., score difference). This example builds on the single-stat validation above:

```typescript
const pair = await discoverScoreRecord(httpClient, 2);
const [statKey, statKey2] = pair.statKeys;
const response2 = await httpClient.get("/scores/stat-validation", {
  params: {
    fixtureId: pair.fixtureId,
    seq: pair.seq,
    statKey,
    statKey2,
  }
});
const validation2 = response2.data;

if (
  Number(validation2.summary.fixtureId) !== pair.fixtureId ||
  Number(validation2.Ts ?? validation2.ts) !== pair.ts ||
  Number(validation2.statToProve.key) !== statKey ||
  Number(validation2.statToProve2.key) !== statKey2
) {
  throw new Error("Two-stat proof does not match the requested record");
}

const fixtureSummary2 = {
  fixtureId: new BN(validation2.summary.fixtureId),
  updateStats: {
    updateCount: validation2.summary.updateStats.updateCount,
    minTimestamp: new BN(validation2.summary.updateStats.minTimestamp),
    maxTimestamp: new BN(validation2.summary.updateStats.maxTimestamp),
  },
  eventsSubTreeRoot: toBytes32(validation2.summary.eventStatsSubTreeRoot),
};
const fixtureProof2 = toProofNodes(validation2.subTreeProof);
const mainTreeProof2 = toProofNodes(validation2.mainTreeProof);
const stat1ForPair = {
  statToProve: validation2.statToProve,
  eventStatRoot: toBytes32(validation2.eventStatRoot),
  statProof: toProofNodes(validation2.statProof),
};
const stat2 = {
  statToProve: validation2.statToProve2,
  eventStatRoot: toBytes32(validation2.eventStatRoot),
  statProof: toProofNodes(validation2.statProof2),
};

const op = { subtract: {} };
const predicate2 = {
  threshold:
    validation2.statToProve.value - validation2.statToProve2.value,
  comparison: { equalTo: {} },
};

const targetTs2 = validation2.summary.updateStats.minTimestamp;
const epochDay2 = Math.floor(targetTs2 / 86400000);
const [dailyScoresPda2] = PublicKey.findProgramAddressSync(
  [
    Buffer.from("daily_scores_roots"),
    new BN(epochDay2).toArrayLike(Buffer, "le", 2),
  ],
  program.programId
);

const isValid2 = await program.methods
  .validateStat(
    new BN(targetTs2),
    fixtureSummary2,
    fixtureProof2,
    mainTreeProof2,
    predicate2,
    stat1ForPair,
    stat2,
    op
  )
  .accounts({
    dailyScoresMerkleRoots: dailyScoresPda2,
  })
  .preInstructions([computeBudgetIx])
  .view();

if (isValid2 !== true) {
  throw new Error("On-chain two-stat validation returned false");
}
console.log("On-chain two-stat validation passed");
```

## V2 Multi-Stat Validation

Use `statKeys` when you need the current V2 payload and strategy model. The requested key order is important: the response arrays `statsToProve` and `statProofs` are mapped by position, and strategy fields such as `index`, `indexA`, `indexB`, and `statIndex` refer to those same `0..N` positions. Every stat in `payload.stats` must be covered exactly once by the strategy, or the program returns `IncompleteStatCoverage`.

```typescript
const multi = await discoverScoreRecord(httpClient, 3);
const requestedKeys = multi.statKeys.slice(0, 3);
const responseV2 = await httpClient.get("/scores/stat-validation", {
  params: {
    fixtureId: multi.fixtureId,
    seq: multi.seq,
    statKeys: requestedKeys.join(","),
  },
});
const validationV2 = responseV2.data;

if (
  Number(validationV2.summary.fixtureId) !== multi.fixtureId ||
  Number(validationV2.Ts ?? validationV2.ts) !== multi.ts ||
  !Array.isArray(validationV2.statsToProve) ||
  !Array.isArray(validationV2.statProofs) ||
  validationV2.statsToProve.length !== requestedKeys.length ||
  validationV2.statProofs.length !== requestedKeys.length
) {
  throw new Error("V2 response shape does not match the request");
}

validationV2.statsToProve.forEach((stat: { key: number }, index: number) => {
  if (Number(stat.key) !== requestedKeys[index]) {
    throw new Error(`V2 stat key at position ${index} is out of order`);
  }
});

const targetTsV2 = validationV2.summary.updateStats.minTimestamp;
const epochDayV2 = Math.floor(targetTsV2 / (24 * 60 * 60 * 1000));

const [dailyScoresPdaV2] = PublicKey.findProgramAddressSync(
  [
    Buffer.from("daily_scores_roots"),
    new BN(epochDayV2).toArrayLike(Buffer, "le", 2),
  ],
  program.programId
);

const payload = {
  ts: new BN(targetTsV2),
  fixtureSummary: {
    fixtureId: new BN(validationV2.summary.fixtureId),
    updateStats: {
      updateCount: validationV2.summary.updateStats.updateCount,
      minTimestamp: new BN(validationV2.summary.updateStats.minTimestamp),
      maxTimestamp: new BN(validationV2.summary.updateStats.maxTimestamp),
    },
    eventsSubTreeRoot: toBytes32(validationV2.summary.eventStatsSubTreeRoot),
  },
  fixtureProof: toProofNodes(validationV2.subTreeProof),
  mainTreeProof: toProofNodes(validationV2.mainTreeProof),
  eventStatRoot: toBytes32(validationV2.eventStatRoot),
  stats: validationV2.statsToProve.map((stat: unknown, index: number) => ({
    stat,
    statProof: toProofNodes(validationV2.statProofs[index]),
  })),
};

const strategy = {
  geometricTargets: [],
  distancePredicate: null,
  discretePredicates: [
    {
      binary: {
        indexA: 0, // requestedKeys[0]
        indexB: 1, // requestedKeys[1]
        op: { subtract: {} },
        predicate: {
          threshold:
            validationV2.statsToProve[0].value
            - validationV2.statsToProve[1].value,
          comparison: { equalTo: {} },
        },
      },
    },
    {
      single: {
        index: 2, // requestedKeys[2]
        predicate: {
          threshold: validationV2.statsToProve[2].value,
          comparison: { equalTo: {} },
        },
      },
    },
  ],
};

const isValidV2 = await program.methods
  .validateStatV2(payload, strategy)
  .accounts({
    dailyScoresMerkleRoots: dailyScoresPdaV2,
  })
  .preInstructions([computeBudgetIx])
  .view();

if (isValidV2 !== true) {
  throw new Error("On-chain V2 validation returned false");
}
console.log("On-chain V2 validation passed");
```

Exercise positional behavior with one key, an ordered pair, the reversed pair, and a supported multi-stat record. A reversed request must produce the same reversed positional mapping; do not sort it after the response arrives. Reject duplicate, empty, negative, or greater-than-u32 keys locally. The hosted OpenAPI defines legacy `statKey`/`statKey2` and V2 `statKeys` as mutually exclusive modes, so do not send both. The live malformed-input status matrix was unavailable without an activated covered bundle; do not claim a particular `4xx` response until it is verified.

## V3 Multiproof Validation

Starting with v1.5.6, the program supports `validateStatV3` — an optimised validation method that uses compressed Merkle multiproofs. The V3 endpoint `/api/scores/stat-validation-v3` accepts up to 5 `statKeys` and returns a `ScoresStatValidationV3` payload containing `multiproof` data (indices and hashes) alongside the familiar fixture and tree proofs.

V3 uses the same `NDimensionalStrategy` as V2 for defining predicates and coverage, but the proof payload is structured differently: instead of individual `statProof` arrays per stat, V3 provides `leaves` (stat data with per-leaf proofs), `leafIndices`, and `multiproofHashes` for compressed verification.

```typescript
const multi = await discoverScoreRecord(httpClient, 3);
const requestedKeys = multi.statKeys.slice(0, 3);
const responseV3 = await httpClient.get("/scores/stat-validation-v3", {
  params: {
    fixtureId: multi.fixtureId,
    seq: multi.seq,
    statKeys: requestedKeys.join(","),
  },
});
const valV3 = responseV3.data;

if (
  Number(valV3.summary.fixtureId) !== multi.fixtureId ||
  !Array.isArray(valV3.statsToProve) ||
  !valV3.multiproof ||
  !Array.isArray(valV3.multiproof.indices) ||
  !Array.isArray(valV3.multiproof.hashes)
) {
  throw new Error("V3 response shape does not match the request");
}

const targetTsV3 = valV3.summary.updateStats.minTimestamp;
const epochDayV3 = Math.floor(targetTsV3 / (24 * 60 * 60 * 1000));

const [dailyScoresPdaV3] = PublicKey.findProgramAddressSync(
  [
    Buffer.from("daily_scores_roots"),
    new BN(epochDayV3).toArrayLike(Buffer, "le", 2),
  ],
  program.programId
);

const payloadV3 = {
  ts: new BN(targetTsV3),
  fixtureSummary: {
    fixtureId: new BN(valV3.summary.fixtureId),
    updateStats: {
      updateCount: valV3.summary.updateStats.updateCount,
      minTimestamp: new BN(valV3.summary.updateStats.minTimestamp),
      maxTimestamp: new BN(valV3.summary.updateStats.maxTimestamp),
    },
    eventsSubTreeRoot: toBytes32(valV3.summary.eventStatsSubTreeRoot),
  },
  fixtureProof: toProofNodes(valV3.subTreeProof),
  mainTreeProof: toProofNodes(valV3.mainTreeProof),
  eventStatRoot: toBytes32(valV3.eventStatRoot),
  leaves: valV3.statsToProve.map((leaf: any) => ({
    stat: leaf.stat,
    statProof: toProofNodes(leaf.statProof),
  })),
  leafIndices: valV3.multiproof.indices,
  multiproofHashes: toProofNodes(valV3.multiproof.hashes),
};

const strategyV3 = {
  geometricTargets: [],
  distancePredicate: null,
  discretePredicates: [
    {
      binary: {
        indexA: 0,
        indexB: 1,
        op: { subtract: {} },
        predicate: {
          threshold:
            valV3.statsToProve[0].stat.value
            - valV3.statsToProve[1].stat.value,
          comparison: { equalTo: {} },
        },
      },
    },
    {
      single: {
        index: 2,
        predicate: {
          threshold: valV3.statsToProve[2].stat.value,
          comparison: { equalTo: {} },
        },
      },
    },
  ],
};

const isValidV3 = await program.methods
  .validateStatV3(payloadV3, strategyV3)
  .accounts({
    dailyScoresMerkleRoots: dailyScoresPdaV3,
  })
  .preInstructions([computeBudgetIx])
  .view();

if (isValidV3 !== true) {
  throw new Error("On-chain V3 validation returned false");
}
console.log("On-chain V3 multiproof validation passed");
```

<Info>
V3 uses the same `NDimensionalStrategy` type as V2. The only difference is the proof payload shape: V3 replaces per-stat `statProofs` with a compressed multiproof structure (`leaves`, `leafIndices`, `multiproofHashes`). All strategy predicates (`single`, `binary`, geometric) work identically.
</Info>

## Common Validation Errors

| Symptom | What to check |
|---------|---------------|
| `InvalidMainTreeProof` | Use the same `targetTs` for `validateStat` and `daily_scores_roots` PDA derivation. Confirm the proof hashes are exactly 32 bytes and not reversed. Confirm fixture ID and `seq` match the returned proof payload. |
| Account integrity passes but proof fails | The PDA is probably on the right network, but the timestamp, interval, or proof payload may not match the on-chain root. Recompute `epochDay = Math.floor(targetTs / 86400000)`. |
| Predicate returns false | First test with an exact predicate against the returned stat value, then replace it with your application predicate. |
| `IncompleteStatCoverage` | Ensure every entry in `payload.stats` (V2) or `payload.leaves` (V3) is referenced exactly once by a discrete predicate or geometric target. Remove unused requested stats or add the missing strategy coverage. |
| V2/V3 strategy validates the wrong stat | Confirm the requested `statKeys` order and the positional strategy indexes match. `index: 0` means the first requested key, not the smallest stat key. |
| `LengthMismatch` (V3) | The `leafIndices` length must match the number of `leaves`. Ensure the multiproof payload was not modified after fetching. |
| `InvalidMultiproof` (V3) | The `multiproofHashes` do not reconstruct the expected event stat root. Verify each hash is exactly 32 bytes and the proof was fetched from the correct endpoint. |
| `MissingProofNode` / `InvalidProofPath` (V3) | A required multiproof node is absent or in the wrong position. Re-fetch the V3 proof and do not reorder or filter any arrays. |
| `401` from the proof endpoint | Renew the guest JWT once and retry with the same `X-Api-Token`. |
| `403` from the proof endpoint | Check the API token, subscription, network, and bundle permission. Do not renew the JWT automatically. |

See [Troubleshooting](/documentation/examples/troubleshooting) for the full activation, streaming, and validation checklist.

## Scores streaming

Use the bounded native-fetch probe in [Streaming Data](/documentation/examples/streaming-data) to discover a score record. An accepted connection or heartbeat proves authentication and transport only. Require a data event for a data-flow pass, and report a 30–45 second accepted window with no covered-fixture event as `inconclusive`.

## Validation Use Cases

On-chain validation enables trustless verification of:

- **Trading Settlement** - Prove score outcomes for bet settlement
- **Conditional Logic** - Execute smart contract logic based on verified game stats
- **Dispute Resolution** - Provide cryptographic proof of game data
- **Automated Markets** - Settle prediction markets with on-chain verification
- **Score Differentials** - Validate margins and score differences for complex betting scenarios
